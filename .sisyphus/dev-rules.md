# 开发规则（通用、可复用）

## 核心规则

### 1. 推送前必须确认 ⚠️

任何代码修改后，推送前必须说：

```
---
⚠️ 推送前确认：
- [ ] 代码已验证可用
- [ ] 用户已确认推送

请确认是否可以推送？
```

用户回复"可以"后才能执行 git push。

### 2. 不确定的问题先问

- 不知道用户想要什么 → 先问
- 不确定某个功能要不要做 → 先问
- 不确定映射关系 → 先问
- 不确定技术上是否可行 → 先问

### 3. 开发完必须测试 ⚠️

- 任何功能开发或 Bug 修复后，必须运行 `npm run build` 进行构建测试
- 构建失败不能推送
- 前端样式/交互修改必须本地验证可用

### 4. 验证后才能说"完成"

- 功能开发 → 部署后让用户验证
- Bug 修复 → 让用户确认问题已解决
- 不能自己说"完成了"就算完成

### 5. 开发流程

```
需求确认 → 编码 → 验证 → 部署 → 用户验收
                  ↑
            这一步不能跳过
```

### 6. 禁止行为

- ❌ 未经确认直接 git push
- ❌ 假设用户会同意就直接执行
- ❌ 不确定的问题跳过不問
- ❌ 未经测试的代码推送到生产
- ❌ 自己说"完成了"就算完成

---

## 简单说

**不要"直接做"，要"先说再问"。用户说"可以"后才能继续。**
